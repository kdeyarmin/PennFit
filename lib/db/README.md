# @workspace/db

Drizzle schema + Postgres connection for the storefront / PennPaps fitter
side of the project. Owns `public.*` tables — orders, products, customers,
admin audit log, and the api-server's auth user table. Used by
`artifacts/api-server` and `artifacts/cpap-fitter`. Migration history is
stored in `drizzle.storefront_migrations` so it does not collide with the
sibling `drizzle.resupply_migrations` table maintained by `lib/resupply-db`.

## Migration policy (ADR 003)

Migrations are checked-in versioned SQL files generated by

```sh
pnpm --filter @workspace/db run generate
```

and applied in production by

```sh
pnpm --filter @workspace/db run migrate
```

(which calls drizzle-orm's `migrate()` from `lib/db/scripts/migrate.mjs`).
The migrator is idempotent and serialized by a `pg_advisory_lock` distinct
from the resupply migrator's, so the two libraries can run concurrently
against the same physical database without blocking each other.

`drizzle-kit push` and `push --force` exist in `package.json` strictly as
developer conveniences against a throwaway dev DB. **Never** point them at
a database that contains real data: `push` diffs the live schema and can
silently rewrite columns. ADR 003 covers the full reasoning.

## Schema-drift guard

`scripts/check-drizzle-drift.sh` snapshots `lib/db/drizzle/`, runs
`pnpm --filter @workspace/db run generate`, diffs the regenerated tree
against the snapshot, and **always** restores the snapshot before
exiting. If `generate` would have produced new SQL or snapshot files,
the check fails — meaning a change to `lib/db/src/schema/*.ts` was
committed without the matching migration. The snapshot is restored on
both the success and failure paths so the check is safe to run
repeatedly without ever leaving a stray randomly-named migration in
the working copy.

When it runs:

- **Pre-commit hook** (`scripts/git-hooks/pre-commit`) — triggered when
  any staged file matches `lib/db/src/schema/*`, `lib/db/drizzle/*`,
  or `lib/db/drizzle.config.ts`. Catches drift before the commit
  lands, so contributors don't find out at validation time minutes
  later.
- **Validation chain** — to wire into the project's `resupply-check`
  validation workflow (defined in `.replit`), append
  ```
   && bash scripts/check-drizzle-drift.sh --self-test \
   && bash scripts/check-drizzle-drift.sh
  ```
  to that workflow's `args` chain. The check is fast and idempotent,
  so it slots in next to the existing `check-codegen` and
  `check-resupply-architecture` invocations without restructuring.

The companion `scripts/check-drizzle-drift.sh.test` exercises the
checker against the live `lib/db/drizzle/` tree (clean-pass scenario
and drifted-schema scenario) so any regression in the drift checker
itself is caught alongside real drift.

The check requires `DATABASE_URL` to be set because `drizzle.config.ts`
reads it on import; without it the script self-skips with a warning so
it is safe to chain into validation steps that may not have a DB
configured.

`lib/resupply-db` is intentionally **not** covered today — its
snapshot meta chain (`drizzle/meta/000N_snapshot.json`) only goes
through 0003 while 25+ migration SQL files have shipped, so
`drizzle-kit generate` short-circuits with a "snapshot collision"
error before it can compute a diff. See the comment block at the top
of `scripts/check-drizzle-drift.sh` for the path to repair the chain
and re-enable the check there.
