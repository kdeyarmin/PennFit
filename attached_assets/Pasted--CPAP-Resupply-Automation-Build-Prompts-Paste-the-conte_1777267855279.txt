## Prompt 1.2 — Eligibility engine

```
In packages/domain, build the EligibilityService — pure functions, no I/O:

- computeEligibility(patient, patientSupply, today) returns:
  { eligible: boolean, eligibleDate: Date, daysUntilEligible: number, reason: string }
- Logic accounts for:
  - Payer rules: Medicare uses LCD allowable schedule; commercial may have plan-specific 
    overrides (read from PatientSupply.commercialOverrideDays); cash-pay has no schedule
  - Pause states: if patient.status != active, or patientSupply.status != active, 
    or pausedUntil > today, not eligible

    until that resolves
  - Compliance: for Medicare PAP supplies, optionally require recent compliance data 
    (configurable lookback window). If no compliance data, flag for clinical review 
    but don't block.
- computeNextOutreachDate(patient, patientSupply, today) — returns when to first reach 
  out (default: 7 days before eligible)
- batchComputeDueList(patients, today) — returns patients due for outreach, with reasons

Property-based tests with fast-check covering edge cases: leap years, time zone 
boundaries, paused patients, terminated insurance, deceased patients (defensive: 
must never reach out).

Document the Medicare LCD references in comments with HCPCS code mappings.
```

## Prompt 7.2.3 — Patient roster importer

```
Build the inbound patient pipeline. This is the most complex importer; it sets the 
pattern for the others.

Process:
1. IngestedFile picked up by ImportProcessor
2. ImportProcessor identifies file type from filename pattern (e.g., 
   patients_YYYYMMDD.csv) and selects the active ImportMapping for that type
3. CSV parsed row-by-row, each row mapped via SchemaMapper to a candidate 
   PatientImportRecord
4. Each record validated against zod schema (E.164 phone, valid email, etc.)
5. All rows staged in PatientImportStaging table with importBatchId
6. After full file is staged, run ReconciliationStep:
   - Match staged rows to existing Patient records via externalId (Pacware's patient ID)
   - Classify each as: NEW (no match), UPDATE (match, fields differ), 
     UNCHANGED (match, no diffs), CONFLICT (match but local changes since last 
     import — see below), MISSING (Patient exists locally but not in this import — 
     could mean deactivated, or could mean this is a partial export)
7. Apply changes within a single transaction:
   - NEW: insert
   - UPDATE: update with field-level audit log
   - CONFLICT: do NOT auto-resolve. Stage as IntegrationConflict for operator review.
   - MISSING: do NOT auto-deactivate from a partial file. Only deactivate if file 
     metadata says "full snapshot" (configurable per import type).
8. ImportBatch finalized with summary: rows in/new/updated/unchanged/conflicts/errors

Conflict detection (critical):
- Each Patient has lastSyncedFromPacwareAt and a hash of synced fields
- If our local copy was modified after last sync (e.g., patient updated address via 
  portal between syncs), and Pacware's value also differs, flag as CONFLICT
- Do not silently overwrite. Operations decides which side wins per case.

Audit:
- Every field change has an audit row with old value, new value, source 
  (pacware_import vs portal vs operator)
- Field-level audit, not just record-level

Tests:
- Full snapshot import to empty DB
```
